# 历史尽调报告查询服务 — 需求文档

## 1. 概述

### 1.1 背景

在现有智能尽调智能体（IDDagent）中增加**历史尽调报告查询**能力，使客户经理能够通过自然对话检索、浏览历史上已创建的尽调报告，查看详情、编辑、下载及管理附件。

### 1.2 目标

- 为用户提供**对话式**历史尽调报告查询入口
- 支持企业名称**模糊输入 + 候选点击**交互
- 以表格形式展示查询结果，并提供详情 / 编辑 / 下载 / 附件操作

---

## 2. 功能需求

### 2.1 技能定义

在现有技能体系（`SkillRegistry`）中注册新技能：

| 属性 | 值 |
|------|-----|
| **技能名称** | `query_historical_dd_reports` |
| **触发关键词** | 历史尽调、尽调记录、历史报告、查一下之前、以往的尽调、历史查询、查看历史、尽调历史、以前的报告 |
| **必需参数** | `company_name`（企业名称）/ `credit_code`（企业编号），至少其一 |
| **可选参数** | `id_type`（证件类型）、`id_number`（证件号码） |
| **特殊参数** | `date_from`（开始日期）、`date_to`（结束日期） |
| **返回类型** | 结构化为 `historical_dd_query_result` 的事件类型 |

### 2.2 交互流程

#### 阶段一：信息收集（对话式补全）

**规则：** 当 `CoordinatorService` 将用户意图路由到 `query_historical_dd_reports` 后，执行器按以下策略提取 / 追问信息：

| 场景 | Coordinator 决策 | ChatController 处理 |
|------|-----------------|-------------------|
| 用户明确提供企业名称 / 编号 + 时间区间 | 直接 `action: "skill"`，params 携带全部参数 | 执行技能并返回表格结果 |
| 用户只说"查询历史尽调报告" | 路由到 skill，params 中携带 `_need_info: true` | 前端通过 SSE 事件 `text_delta` 返回追问文本："请问您要查询哪家企业的历史尽调报告？" |
| 用户提供了企业名但无时间区间 | 路由到 skill，params 携带 `company_name` | 技能返回 `action: "need_date_range"`，追问："请选择尽调申请的时间区间（开始日期 - 结束日期）" |
| 用户提供了企业名 + 时间区间 | 路由到 skill，params 携带已有字段 | 执行查询并同时追问（可选）证件类型和号码 |

**追问的对话流示例如下：**

```
用户：我要查询历史尽调报告
助手：请问您要查询哪家企业的历史尽调报告？（可提供企业名称或企业编号）

用户：我想查一下北京星河科技有限公司
助手：请选择尽调申请的时间区间（格式：开始日期 - 结束日期）

用户：2025-01-01 到 2025-12-31
助手：是否提供证件类型和号码以便精确匹配？（可选）
[用户可选择跳过]
```

#### 阶段二：企业名称模糊匹配

**实现要点：**

- 复用 `RiskCheckSkill` 中的 `fuzzyMatchCompany()` / `resolveCompanyMatch()` 模糊匹配引擎
- 匹配到的候选企业名列表以 **SSE 事件 `company_name_candidates`** 返回前端
- 前端渲染为**可点击的候选按钮**（参考现有 `FollowUpChip` 组件样式），用户点击后自动触发规范化查询语句

**候选企业数据结构（SSE 事件）：**

```json
{
  "type": "company_name_candidates",
  "data": {
    "message": "找到以下匹配的企业，请选择：",
    "options": [
      { "credit_code": "91110108MA01B3XK2P", "company_name": "北京星河科技有限公司" },
      { "credit_code": "91440300MA5DCTJH8N", "company_name": "北京星河供应链管理有限公司" }
    ]
  }
}
```

**用户点击候选企业后的行为：**

1. 前端自动在输入框中填入规范化的查询语句，如：
   > 查询历史尽调报告，企业：北京星河科技有限公司（91110108MA01B3XK2P）
2. 自动发起该消息请求（等同于用户手动发送）
3. 服务端收到完整参数后直接执行查询

#### 阶段三：查询结果展示

查询结果以 `historical_dd_query_result` 事件返回，前端渲染为表格。

**SSE 事件格式：**

```json
{

  "data": {
    "action": "result",
    "company_name": "北京星河科技有限公司",
    "credit_code": "91110108MA01B3XK2P",
    "total_count": 3,
    "query_params": {
      "date_from": "2025-01-01",
      "date_to": "2025-12-31"
    },
    "records": [
      {
        "id": "DD-2025-001",
        "institution": "中国工商银行北京分行",
        "credit_code": "91110108MA01B3XK2P",
        "company_name": "北京星河科技有限公司",
        "status": "completed",
        "status_label": "创建成功",
        "created_at": "2025-03-15",
        "updated_at": "2025-03-20"
      },
      {
        "id": "DD-2025-002",
        "institution": "中国工商银行北京分行",
        "credit_code": "91110108MA01B3XK2P",
```

### 2.3 结果表格设计

前端渲染表格，包含以下五列：

| 列名 | 内容 | 说明 |
|------|------|------|
| **序号** | 1, 2, 3... | 从 1 开始的连续编号 |
| **机构** | `institution` | 尽调报告所属机构名称 |
| **统一社会信用代码** | `credit_code` | 企业信用代码 |
| **报告状态** | `status_label` | **创建成功**（绿色标签）或 **未完成**（灰色标签）|
| **操作** | 四个按钮 | 查看详情、编辑、下载、附件 |

**操作按钮行为：**

| 按钮 | 行为 | 适用状态 |
|------|------|---------|
| **查看详情** | 跳转到 `h5/dd-report-detail.html?report_id={id}`，展示完整尽调报告 | 创建成功 / 未完成 |
| **编辑** | 跳转到 `h5/dd-report-edit.html?report_id={id}`，进入编辑模式 | 创建成功 / 未完成 |
| **下载** | 触发 `GET /api/dd-reports/{id}/download`，返回 PDF/Word 文件 | 仅创建成功 |
| **附件** | 跳转到 `h5/dd-report-attachments.html?report_id={id}`，管理报告附件（上传 / 查看 / 删除） | 创建成功 / 未完成 |

### 2.4 历史尽调报告详情页

**查看详情** 跳转到 H5 页面后展示的内容（参考现有 `risk-report.html` 设计风格）：

- **报告头部**：报告编号、企业名称、信用代码、报告状态、创建时间
- **基本信息**：企业名称、统一社会信用代码、注册地址、注册资本、法定代表人
- **尽调信息**：
  - 开户目的
  - 资金来源
  - 预期交易规模
  - 跨境业务是否涉及
  - 风险评级
- **风险信息**：关联到 `check_company_risk` 的风险结论摘要
- **产品签约信息**：网银、对账、短信等签约情况
- **审核结论**：尽调结论文字
- **附件列表**：关联的附件文件

### 3.2 前端新增类型（`types/index.ts`）

```typescript
// 历史尽调报告记录
export interface HistoricalDDReport {
  id: string;
  institution: string;
  credit_code: string;
  company_name: string;
  status: 'completed' | 'incomplete';
  status_label: string;
  created_at: string;
  updated_at: string;
}

// 历史尽调查询结果
export interface HistoricalDDQueryResult {
  action: 'result' | 'need_date_range' | 'not_found';
  company_name?: string;
  credit_code?: string;
  total_count: number;
  query_params?: {
    date_from?: string;
    date_to?: string;
  };
  records?: HistoricalDDReport[];
  message?: string;
}

// 企业名称候选
export interface CompanyNameCandidate {
  credit_code: string;
  company_name: string;
}

// 企业名称候选事件
export interface CompanyNameCandidatesEvent {
  action: 'candidates';
  message: string;
  options: CompanyNameCandidate[];
}
```

**SSE 事件类型新增：**

```typescript
export type SSEEventType =
  // ... 现有类型 ...
  | 'historical_dd_query_result'
  | 'company_name_candidates'
  | 'need_date_range';
```

## 4. 后端架构

### 4.1 新增文件

| 文件 | 功能 |
|------|------|
| `skill/HistoricalDDQuerySkill.java` | 历史尽调查询技能，注册到 SkillRegistry，处理参数补全 / 模糊匹配 / 查询执行 |
| `controller/HistoricalDDReportController.java` | REST API 控制器：报告详情查询、文件下载、附件管理等 |
| `service/DDReportService.java` | 报告数据服务：CRUD、文件下载、索引管理（读写 JSON 文件 + 并发安全） |

### 4.2 API 接口

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/dd-reports/{id}` | 获取报告详情 JSON |
| `GET` | `/api/dd-reports/{id}/download` | 下载报告文件（PDF/Word） |
| `PUT` | `/api/dd-reports/{id}` | 更新报告内容（编辑保存） |
| `GET` | `/api/dd-reports/{id}/attachments` | 获取报告附件列表 |
| `POST` | `/api/dd-reports/{id}/attachments` | 上传报告附件 |
| `DELETE` | `/api/dd-reports/{id}/attachments/{fileId}` | 删除报告附件 |

### 4.3 存储设计

**无数据库，沿用项目现有 JSON 文件存储模式。** 使用 `ConcurrentHashMap` + `synchronized save()` 确保并发安全，参考 `ConversationService` 和 `UserStoreService` 的实现模式。

## 6. 非功能需求

### 6.1 权限安全

- 所有 `/api/dd-reports/*` 接口需 JWT 认证（通过 `JwtAuthFilter`）
- 用户仅能查看自己所属机构的报告
- 附件下载链接使用 UUID 防止路径穿越（参考 `ChatAttachmentController` 实现）

### 6.2 状态管理

- 报告状态分为 `completed`（创建成功）和 `incomplete`（未完成）
- 状态变更写回 JSON 文件时需加 `synchronized` 锁

### 6.3 模糊匹配复用

- 企业名称模糊匹配直接复用 `RiskCheckSkill` 中的 `fuzzyMatchCompany()` 和 `resolveCompanyMatch()` 方法
- 在 `DataLoader` 中构建统一的企业名称索引访问

### 6.4 Coordinator 提示词更新

在 `CoordinatorService.buildSystemPrompt()` 中新增第 8 条路由规则：

```
8. 当用户输入中包含"历史尽调"、"查询历史"、"尽调记录"、"历史报告"、"以前的报告"等关键词时，
   必须匹配为 query_historical_dd_reports 技能。
```
